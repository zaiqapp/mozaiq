# Mozaiq v2.4 â€” Widget Property Editor & New Widget Types Design

> **For agentic workers:** Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task.

**Goal:** Add a schema-driven property editor to the right panel and introduce 9 new multi-series/comparison widget types, enabling users to edit widget config directly and giving AI more chart types to generate meaningful comparisons.

**Architecture:** A new `widget-config-schema.ts` defines config form fields per widget type. A new `WidgetConfigPanel` renders those fields. The right panel gains two tabs (Properties | Data). Nine new Recharts-based widget components are registered with field definitions and AI prompt entries.

**Tech Stack:** Next.js App Router, Recharts, Zustand, TypeScript, Tailwind CSS, shadcn/ui

## Design Decisions

### Right panel layout
Two tabs: **Properties** and **Data**.
- Properties tab â€” `WidgetConfigPanel`: schema-driven form for all config fields
- Data tab â€” existing `WidgetMappingPanel`: data source connection and axis mapping (unchanged)

The **existing inline title/description/color fields** currently rendered directly in `RightPanel.tsx` are **removed** and replaced entirely by the schema-driven Properties tab. The schema includes `title` and `description` for all widget types.

### Property editor
- Config fields per widget type defined in `lib/widget-config-schema.ts` as `ConfigFieldDef[]`
- Field types: `text`, `number`, `color`, `boolean`, `select`
- All widgets share two common fields: `title` (text) and `description` (text, multiline)
- `text-note` additionally includes `content` (text, multiline) as it is the widget's primary field
- Widget-specific fields listed below in the Config Schema section
- Edits dispatch `updateWidgetConfig` (existing Zustand action)

**`WidgetConfigPanel` empty state:** When no widget is selected, `RightPanel` already renders nothing (the panel is hidden). `WidgetConfigPanel` can assume `widget` is always a valid `Widget` object â€” no null guard needed inside the component.

**`widgetConfigSchema` TypeScript note:** Typed as `Record<WidgetType, ConfigFieldDef[]>` â€” this is an intentional compile-time guard. Any new type added to the `WidgetType` union must also be added here or TypeScript will error at the record literal. Do not relax this to `Partial<Record<...>>`.

**`WidgetConfigPanel` prop signature:**
```tsx
interface WidgetConfigPanelProps {
  widget: Widget  // full widget object (id, type, config, dataSourceId, dataSourceMapping)
}
```
It reads `widget.config`, `widget.dataSourceMapping`, and `widget.type`. It dispatches `updateWidgetConfig(widget.id, patch)` and `updateWidgetDataSourceMapping(widget.id, newMapping)` from the store.

### Override toggle for data-sourced scalar fields
The override toggle applies **only to scalar config fields** (type: `text`, `number`, `color`, `boolean`, `select`). It does **not** apply to the `data` array-of-objects field on chart widgets â€” that is handled entirely through the Data tab mapping.

When a scalar config field `key` is currently supplied by `dataSourceMapping[key]`:
1. Render field as read-only with **"sourced from [col] Â· override"** label â€” `col` is `dataSourceMapping[key].column` (the `AxisMapping.column` property)
2. On override click: dispatch `updateWidgetDataSourceMapping(widgetId, omit(mapping, key))` â€” removes just that entry
3. Field becomes editable; typed value saved via `updateWidgetConfig`

**Mapping key convention:** The `dataSourceMapping` key matches `ConfigFieldDef.key` for scalar fields. For example, KPI's `value` field is at `dataSourceMapping['value']`; stat-comparison's primary is at `dataSourceMapping['primary']`. This is already the convention established by the existing field registry (fieldKey === mapping key === config key for scalar fields).

### Multi-series charts â€” data flow
Multi-series charts (`multi-line-chart`, `grouped-bar-chart`, `stacked-bar-chart`) use **two series** â€” `series1` (required) and `series2` (optional). This covers the primary comparison use case (budgeted vs actual, this vs last period). More series can be added in a future iteration.

Data field registry: one `data` field of type `array-of-objects` (same as existing charts).
Mapping shape: `{ name: { column: "Client" }, series1: { column: "Budget" }, series2: { column: "Actual" } }`
`applyMapping("data", "array-of-objects", rows, mapping)` â†’ `[{ name: "Acme", series1: 50000, series2: 43000 }, ...]` (existing rename mode handles this automatically).

Series labels: stored in `config.seriesLabels: Record<string, string>` (e.g. `{ series1: "Budgeted", series2: "Actual" }`). The AI populates these from the column names. Users can rename them in the Properties tab via two `text` config fields: `seriesLabels.series1` and `seriesLabels.series2`.

**`apply-mapping.ts` â€” no change needed for rename mode.** Rename mode is the default path for any `array-of-objects` field whose key is not `'rows'`. New widget types with `key: 'data'` automatically get rename mode.

**`WidgetRenderer.tsx` â€” do NOT add new types to `CHART_TYPES`.** `WidgetRenderer` uses `CHART_TYPES.has(widget.type)` to gate a `patch['dataKey'] = 'value'` write for single-series chart compatibility. Multi-series types do not use `dataKey` at all. Keep `CHART_TYPES` as-is (existing 5 chart types only). For the 9 new types the mapped data array is passed directly to the component without any `dataKey` patch.

### Scatter chart â€” data flow
`scatter-chart` uses the same pattern as other charts: one `data` field of type `array-of-objects`.
Mapping shape: `{ x: { column: "Budget" }, y: { column: "Hours" }, label: { column: "Client" } }` (label is optional â€” if absent, tooltip shows coordinates only).
Output: `[{ x: 50000, y: 200, label: "Acme" }, ...]`

### Combo chart â€” data shape
Mapping: `{ name: { column: "Client" }, bar: { column: "Revenue" }, line: { column: "Growth%" } }`
`applyMapping` rename mode output: `[{ name: "Acme", bar: 48000, line: 12.4 }, ...]`
The component renders a `<ComposedChart>` with one `<Bar dataKey="bar">` and one `<Line dataKey="line">`.

### Waterfall chart â€” Recharts implementation note
Recharts has no native `<WaterfallChart>`. The standard approach is to use a `<BarChart>` with two `<Bar>` series stacked:
1. An invisible "base" bar (opacity 0) that positions each visible bar at the right cumulative height
2. A visible "delta" bar colored green (positive) or red (negative)
The component must pre-process `data` rows to compute running totals and split into base + delta arrays before rendering.

## Multi-Series Rendering Pattern

```tsx
const SERIES_COLORS = ['#6366f1', '#10b981', '#f59e0b', '#ef4444']
const seriesKeys = ['series1', 'series2'].filter(k => data[0]?.[k] !== undefined)

return (
  <BarChart data={data}>
    <XAxis dataKey="name" />
    <YAxis />
    <Tooltip />
    <Legend />
    {seriesKeys.map((key, i) => (
      <Bar
        key={key}
        dataKey={key}
        name={config.seriesLabels?.[key] ?? key}
        fill={SERIES_COLORS[i]}
      />
    ))}
  </BarChart>
)
```

## Stat Comparison â€” Data Source Behavior

`stat-comparison` uses `type: 'number'` field defs for `primary` and `secondary`. `applyMapping` sums all rows for number-type fields (as of v2.3 fix). This is intentional: "Total budgeted hours across all clients" vs "Total actual hours" is the primary use case. If a user's data has one summary row, summing that single row is a no-op. If they have multi-row data (one row per client), summing gives the correct aggregate total.

## Stat Comparison Delta Calculation

```ts
const delta = primary !== 0 ? ((secondary - primary) / Math.abs(primary)) * 100 : null
const isPositive = delta !== null && delta >= 0
// if deltaPositiveIsGood === false, invert color meaning
const isGood = config.deltaPositiveIsGood ? isPositive : !isPositive
```

## AI Prompt Additions

New types appended to `GENERATE_WIDGETS_SYSTEM_PROMPT`:

```
- "stat-comparison": { "primary": { "column": "<numeric col>" }, "secondary": { "column": "<numeric col>" } } â€” use for head-to-head comparisons. Config: { "primaryLabel": "Budgeted", "secondaryLabel": "Actual" }
- "multi-line-chart": { "name": { "column": "<x col>" }, "series1": { "column": "<y col>" }, "series2": { "column": "<y col>" } } â€” up to 2 series; use when comparing trends of 2 metrics over time
- "grouped-bar-chart": { "name": { "column": "<category>" }, "series1": { "column": "<val>" }, "series2": { "column": "<val>" } } â€” use for side-by-side comparison per category. Config: { "seriesLabels": { "series1": "Budget", "series2": "Actual" } }
- "stacked-bar-chart": same mapping as grouped-bar-chart â€” use when part-to-whole breakdown matters
- "scatter-chart": { "x": { "column": "<numeric>" }, "y": { "column": "<numeric>" }, "label": { "column": "<category>" } } â€” use for correlation between two numeric columns
- "combo-chart": { "name": { "column": "<category>" }, "bar": { "column": "<numeric>" }, "line": { "column": "<numeric>" } } â€” use when one metric is volume and another is a rate/ratio. Config: { "barLabel": "Revenue", "lineLabel": "Growth %" }
- "ranked-list": { "label": { "column": "<category>" }, "value": { "column": "<numeric>" } } â€” use for top-N rankings; simpler than bar-chart
- "waterfall-chart": { "label": { "column": "<category>" }, "value": { "column": "<numeric>" } } â€” use for sequential cumulative breakdowns (revenue components, budget categories)
- "treemap": { "name": { "column": "<category>" }, "value": { "column": "<numeric>" } } â€” use for proportional share when there are many categories
```

Updated rules:
- Prefer `stat-comparison` over two separate KPIs when comparing two metrics directly
- Prefer `multi-line-chart` or `grouped-bar-chart` over separate single-series charts for multi-metric comparisons
- Prefer `ranked-list` over `bar-chart` for simple top-N use cases
- When generating `multi-line-chart`, `grouped-bar-chart`, or `stacked-bar-chart`, always populate `config.seriesLabels` with human-readable names from the column names

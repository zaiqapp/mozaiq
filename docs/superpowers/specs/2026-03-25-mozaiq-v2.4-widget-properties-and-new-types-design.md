# Mozaiq v2.4 ‚Äî Widget Property Editor & New Widget Types Design

> **For agentic workers:** Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task.

**Goal:** Add a schema-driven property editor to the right panel and introduce 9 new multi-series/comparison widget types, enabling users to edit widget config directly and giving AI more chart types to generate meaningful comparisons.

**Architecture:** A new `widget-config-schema.ts` defines config form fields per widget type. A new `WidgetConfigPanel` renders those fields. The right panel gains two tabs (Properties | Data). Nine new Recharts-based widget components are registered with field definitions and AI prompt entries.

**Tech Stack:** Next.js App Router, Recharts, Zustand, TypeScript, Tailwind CSS, shadcn/ui

---

## Design Decisions

### Right panel layout
Two tabs: **Properties** and **Data**.
- Properties tab ‚Äî `WidgetConfigPanel`: schema-driven form for all config fields
- Data tab ‚Äî existing `WidgetMappingPanel`: data source connection and axis mapping (unchanged)

### Property editor
- Config fields per widget type defined in `lib/widget-config-schema.ts` as `ConfigFieldDef[]`
- Field types: `text`, `number`, `color`, `boolean`, `select`
- All widgets share common fields: `title`, `description`
- Widget-specific fields: KPI gets `prefix`, `suffix`, `changeLabel`; gauge gets `min`, `max`, `unit`; stat-comparison gets `primaryLabel`, `secondaryLabel`; etc.
- Edits dispatch `updateWidgetConfig` (existing Zustand action)

### Override toggle for data-sourced fields
When a numeric/string config field is currently being supplied by a data source mapping:
- Show the field as read-only with a **"sourced from [col] ¬∑ override"** link
- Clicking override removes that specific mapping entry from `dataSourceMapping` and makes the field editable
- User types a manual value which is saved directly to config
- This allows fixing incorrect AI-generated values without disconnecting the entire data source

### Multi-series data mapping
Multi-series charts (`multi-line-chart`, `grouped-bar-chart`, `stacked-bar-chart`) use up to 4 named series:
- `name` ‚Üí x-axis / category column (required)
- `series1` ‚Üí first value column (required)
- `series2`, `series3`, `series4` ‚Üí additional value columns (optional)
- The `dataSourceMapping` stores labels alongside columns: `{ series1: { column: "Revenue", displayName: "Revenue" }, ... }`
- `applyMapping` rename mode outputs `[{ name: "Jan", series1: 100, series2: 80 }, ...]`
- Components render one `<Line>` / `<Bar>` per present series key

### AI prompt updates
All 9 new types added to `GENERATE_WIDGETS_SYSTEM_PROMPT` with field shapes and usage guidance. AI is instructed to prefer multi-series types for comparisons rather than generating separate single-series charts.

## Config Field Schema

```ts
// lib/widget-config-schema.ts
export type ConfigFieldType = 'text' | 'number' | 'color' | 'boolean' | 'select'

export interface ConfigFieldDef {
  key: string           // matches key in widget config interface
  label: string
  type: ConfigFieldType
  options?: { value: string; label: string }[]  // for 'select' type
  placeholder?: string
  min?: number
  max?: number
  step?: number
}

export type WidgetConfigSchema = ConfigFieldDef[]

// Example for KPI:
// [
//   { key: 'title', label: 'Title', type: 'text' },
//   { key: 'prefix', label: 'Prefix', type: 'text', placeholder: '$' },
//   { key: 'suffix', label: 'Suffix', type: 'text', placeholder: '%' },
//   { key: 'changeLabel', label: 'Change label', type: 'text', placeholder: 'vs last month' },
//   { key: 'color', label: 'Color', type: 'color' },
// ]
```

## Override Toggle Behaviour

`WidgetConfigPanel` needs access to both `widget.config` and `widget.dataSourceMapping`. For each `ConfigFieldDef`:

1. Check if `dataSourceMapping[field.key]` exists
2. If yes: render field as read-only with `"sourced from [column] ¬∑ override"` label
3. On override click: dispatch `updateWidgetDataSourceMapping(widgetId, { ...mapping, [field.key]: undefined })` to remove just that entry
4. Field becomes editable; user value saved via `updateWidgetConfig`

## Multi-Series Rendering Pattern

```tsx
// GroupedBarChart example
const SERIES_COLORS = ['#6366f1', '#10b981', '#f59e0b', '#ef4444']
const seriesKeys = ['series1', 'series2', 'series3', 'series4'].filter(
  k => data[0]?.[k] !== undefined
)

return (
  <BarChart data={data}>
    <XAxis dataKey="name" />
    <YAxis />
    <Tooltip />
    <Legend />
    {seriesKeys.map((key, i) => (
      <Bar key={key} dataKey={key} name={config.seriesLabels?.[key] ?? key} fill={SERIES_COLORS[i]} />
    ))}
  </BarChart>
)
```

Series labels come from `config.seriesLabels` (a `Record<string, string>` the AI populates from column names) and are editable in the Properties tab.

## Stat Comparison Delta Calculation

```ts
const delta = primary !== 0 ? ((secondary - primary) / Math.abs(primary)) * 100 : null
const isPositive = delta !== null && delta >= 0
```

Direction interpretation (positive = good or bad) is set by a `deltaPositiveIsGood: boolean` config field, defaulting to `true`. When `false`, positive delta renders red (e.g. "hours over budget").

## AI Prompt Additions

New types appended to `GENERATE_WIDGETS_SYSTEM_PROMPT`:

```
- "stat-comparison": { "primary": { "column": "<numeric col>" }, "secondary": { "column": "<numeric col>" } } ‚Äî use for head-to-head comparisons (budgeted vs actual, this vs last period)
- "multi-line-chart": { "name": { "column": "<x col>" }, "series1": { "column": "<y col>" }, "series2": { "column": "<y col>" } } ‚Äî up to series4; use when comparing trends of multiple metrics over time
- "grouped-bar-chart": { "name": { "column": "<category>" }, "series1": { "column": "<val>" }, "series2": { "column": "<val>" } } ‚Äî use for side-by-side comparison per category
- "stacked-bar-chart": same mapping as grouped-bar-chart ‚Äî use when part-to-whole breakdown matters
- "scatter-chart": { "x": { "column": "<numeric>" }, "y": { "column": "<numeric>" } } ‚Äî use for correlation between two numeric columns
- "combo-chart": { "name": { "column": "<category>" }, "bar": { "column": "<numeric>" }, "line": { "column": "<numeric>" } } ‚Äî use when one metric is volume and another is a rate/ratio
- "ranked-list": { "label": { "column": "<category>" }, "value": { "column": "<numeric>" } } ‚Äî use for top-N rankings; simpler than bar-chart
- "waterfall-chart": { "label": { "column": "<category>" }, "value": { "column": "<numeric>" } } ‚Äî use for sequential cumulative breakdowns (revenue components, budget categories)
- "treemap": { "name": { "column": "<category>" }, "value": { "column": "<numeric>" } } ‚Äî use for proportional share when there are many categories
```

Updated rules:
- Prefer `stat-comparison` over two separate KPIs when comparing two metrics
- Prefer `multi-line-chart` or `grouped-bar-chart` over separate single-series charts for multi-metric comparisons
- Prefer `ranked-list` over `bar-chart` for simple top-N use cases

# Mozaiq v2.1 Design Spec — Auth + User Accounts

**Date:** 2026-03-25
**Status:** Approved
**Scope:** Authentication, user ownership of dashboards, My Dashboards page, navigation updates, AI generation rate limiting

---

## Overview

v2.1 introduces user accounts to mozaiq. Until now, dashboards are anonymous and ownerless. v2.1 adds Clerk-based auth (GitHub + Google OAuth), associates dashboards with users, and gives users a `/my-dashboards` page to manage their work.

The builder stays frictionless — anyone can build without logging in. Auth is only required at the point of saving.

This is the first of three v2 sub-releases:
- **v2.1** — Auth + User Accounts (this spec)
- **v2.2** — Real Data Sources (Google Sheets, CSV upload)
- **v2.3** — AI Enhancements (per-widget AI, chart recommendations, dashboard insights)

---

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Auth provider | Clerk | Pre-built OAuth UI, Vercel-native, zero session wiring |
| OAuth providers | GitHub + Google | Covers the target developer/analyst audience |
| Existing anonymous dashboards | Left as orphaned | All existing dashboards are test data, no migration needed |
| Builder auth gate | Build first, login to save | Maximises top-of-funnel conversion; Zustand holds state client-side |
| User storage | No local User table | `clerkUserId` stored directly as `userId` on Dashboard; Clerk is source of truth |
| Auth approach | Client-side check before save (Approach C) | Best UX (no wasted API round-trip), server middleware as backstop |
| AI rate limiting | Lifetime cap via GenerationLog | Uses existing table; limit controlled by `AI_GENERATION_LIMIT` env var (default: 2) for anonymous (by IP) and free authenticated users (by userId) |

### Required schema changes

1. **Drop the `User` relation from `Dashboard`** — The existing schema has `user User? @relation(fields: [userId], references: [id])` which expects `userId` to be a Prisma-managed `cuid()`. Clerk user IDs (e.g. `user_abc123`) will violate this foreign key constraint. The relation lines must be removed from both `Dashboard` and `User` models in a migration before Clerk IDs can be stored. `userId String?` remains as a plain nullable string field.

2. **Drop `model User` and `enum Plan` from `schema.prisma`** — Once the relation is removed, the `User` model and the `Plan` enum it references become dead code. Clerk is the source of truth for user identity; plan management will be handled via Clerk metadata in v3. Remove both to keep the schema clean.

3. **Add `ip String?` to `GenerationLog`** — for anonymous rate limiting by IP address.

4. **Add `userId String?` to `GenerationLog`** — for authenticated user rate limiting. (The field does not currently exist despite an earlier incorrect claim in this doc.)

## Feature Details

### 1. Clerk Setup

- Install `@clerk/nextjs`
- Add to Vercel project via Marketplace → auto-provisions `CLERK_SECRET_KEY` and `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`
- Manually set: `NEXT_PUBLIC_CLERK_SIGN_IN_URL=/sign-in` and `NEXT_PUBLIC_CLERK_SIGN_UP_URL=/sign-up`
- Do **not** set `NEXT_PUBLIC_CLERK_AFTER_SIGN_IN_URL` or `NEXT_PUBLIC_CLERK_AFTER_SIGN_UP_URL` — leaving both unset keeps the modal in-place so the builder's `useEffect` retry fires correctly. Setting either redirect URL would navigate the user away from the builder after sign-in/up, breaking the auto-retry flow.
- `<ClerkProvider>` wraps `app/layout.tsx`
- Sign-in and sign-up pages use Clerk's `<SignIn />` and `<SignUp />` components, centered on a dark page matching the app theme

### 2. Middleware (`proxy.ts`)

Protects:
- `/my-dashboards` — redirect to `/sign-in` if unauthenticated
- `PATCH /api/dashboards/[id]` — return 401 if no session
- `DELETE /api/dashboards/[id]` — return 401 if no session
- `GET /api/dashboards` — return 401 if no session

Does **not** protect:
- `/builder` and `/builder/[id]` — public
- `GET /api/dashboards/[id]` — public (powers share view)
- `POST /api/dashboards` — public (allows anonymous create; `userId` extracted server-side from Clerk session via `auth()` if present — never trusted from request body)
- `POST /api/ai/generate` — public (rate-limited instead)

### 3. Save Flow (Builder)

**Unauthenticated path:**
1. User clicks Save
2. `Toolbar` reads `useAuth().isSignedIn` → `false`; calls `saveDashboard(false)`
3. `saveDashboard()` receives `isSignedIn = false`; calls `openSignIn()` (from `useClerk()`, passed in or accessed via a ref)
4. `useEffect` in `Toolbar` watches `isSignedIn` — fires `saveDashboard(true)` automatically when it transitions to `true`
5. API called with active Clerk session → `auth()` server-side extracts `userId` → dashboard saved with `userId`

**Authenticated path:**
1. User clicks Save
2. `Toolbar` reads `useAuth().isSignedIn` → `true`; calls `saveDashboard(true)`
3. API called immediately with Clerk session → dashboard saved with `userId`

**Note:** `useAuth()` and `useEffect` live in `Toolbar.tsx` (a React component), not in the Zustand store. Zustand store actions are plain JS and cannot call React hooks.

The Save button shows a subtle `↗` arrow indicator when the user is unauthenticated to hint that sign-in is required. No tooltip or banner needed.

### 4. Navigation

**TopNav** — shown on: landing page (`/`), share view (`/dashboard/[id]`), my-dashboards (`/my-dashboards`)

- Left: mozaiq logo → links to `/`
- Right (unauthenticated): "Sign In" button → opens Clerk modal
- Right (authenticated): "My Dashboards" link + `<UserMenu>` avatar

**Builder Toolbar** — no TopNav (would double-stack with existing toolbar)

- Far right: `<UserMenu>` avatar when authenticated; empty slot when not
- Same `<UserMenu>` dropdown component as TopNav

**`<UserMenu>` dropdown (shared component):**
- User email
- Plan tier (e.g. "Free plan")
- "My Dashboards" → navigates to `/my-dashboards`
- "Sign Out" → Clerk `signOut()`

### 5. `/my-dashboards` Page

Server Component, fetches dashboards via Prisma where `userId = clerkUserId`, ordered by `updatedAt` desc.

**Header row:**
- Title: "My Dashboards"
- Subtitle: "{n} dashboards"
- "New Dashboard" button → `/builder`

**Dashboard card (hybrid list):**
- Mini layout preview (colour-coded widget grid, 56×40px)
- Dashboard name
- Widget count + last edited date
- Actions: Edit → `/builder/[id]`, Share (copies share URL to clipboard), ⋯ overflow menu

**⋯ overflow menu:**
- Duplicate — POSTs a new dashboard with the same layout and widgets, name prefixed with "Copy of ", and the current user's `clerkUserId` as `userId`
- Rename — inline edit of the dashboard name, saved via `PATCH /api/dashboards/[id]`
- Delete — confirmation dialog, then `DELETE /api/dashboards/[id]`

**Empty state:**
- Dashed border card, centred
- Heading: "No dashboards yet"
- Body: "Build your first dashboard or let AI generate one from a prompt."
- CTA: "Create your first dashboard →" → `/builder`

### 6. API Ownership

| Route | Method | Behaviour |
|-------|--------|-----------|
| `/api/dashboards` | `POST` | Extracts `clerkUserId` from session (if present); saves as `userId`. Anonymous creates set `userId = null`. |
| `/api/dashboards` | `GET` | Requires auth. Returns only dashboards where `userId = clerkUserId`. |
| `/api/dashboards/[id]` | `GET` | Public. No auth check. Powers share view. |
| `/api/dashboards/[id]` | `PATCH` | Requires auth. Returns 403 if `dashboard.userId !== clerkUserId`. |
| `/api/dashboards/[id]` | `DELETE` | Requires auth. Returns 403 if `dashboard.userId !== clerkUserId`. |

### 7. AI Generation Rate Limiting

Enforced in `POST /api/ai/generate` before calling the model:

**Anonymous users (no Clerk session):**
- Count rows in `GenerationLog` where `ip = requestIp` and `success = true`
- If count ≥ limit → return 429 with message: "You've used your free AI generations. Sign up to get more."

**Authenticated free users:**
- Count rows in `GenerationLog` where `userId = clerkUserId` and `success = true`
- If count ≥ limit → return 429 with message: "You've used your free AI generations. Upgrade to Pro for unlimited."

**Limit:** Controlled by `AI_GENERATION_LIMIT` env var (default: `2`). Anonymous and authenticated free users share the same limit value. Pro plan (v3) removes the cap entirely — implement as a simple `if (isPro) skip check` guard when billing ships.

**Implementation note:** The rate limit check must be inserted **before** the `generateText` call in the route handler. The existing `finally`-block log must also be updated to write `ip` and `userId` to `GenerationLog` alongside the existing fields.

## Out of Scope for v2.1

- Billing / plan enforcement (v3)
- Team workspaces (v4)
- Dashboard sharing permissions (public/private toggle) — all saved dashboards remain publicly accessible via share link
- Email verification or magic link auth
- User profile page
- Dashboard thumbnails (mini previews are abstract colour blocks, not real screenshots)
- `isPublic` toggle — the `isPublic Boolean @default(true)` field on `Dashboard` remains at its default for all dashboards in v2.1 and is not surfaced in the UI. All saved dashboards are publicly accessible via share link.
- `TopNav` uses `position: static` on the share view (`/dashboard/[id]`) to preserve iframe embed behaviour — no fixed/sticky positioning that would cause layout issues inside an iframe.
- Clipboard fallback — the Share action in `DashboardCard` uses `navigator.clipboard.writeText()` (HTTPS/localhost only). Since mozaiq runs on Vercel (HTTPS), no fallback is required; a "Copied!" toast confirms success.

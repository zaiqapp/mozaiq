# Mozaiq v2.3 ‚Äî Global Data Sources & AI Widget Generation

## Overview

Replace the per-widget data source model with dashboard-level global data sources. Users connect CSV or Google Sheets once at the dashboard level, then AI auto-generates appropriate widgets from the data. Per-widget mapping gains axis-aware controls and optional display name overrides.

---

## Data Model

### New type: `GlobalDataSource`

```ts
export interface GlobalDataSource {
  id: string                              // nanoid
  name: string                            // user-assigned, e.g. "Sales Q1"
  type: 'csv' | 'google-sheets'
  data?: Record<string, unknown>[]        // CSV inline data
  fileName?: string                       // CSV original filename
  url?: string                            // Google Sheets URL
  gid?: string                            // Sheet tab ID
  refreshInterval?: number                // seconds; 0 = manual
}
```

### New type: `AxisMapping`

```ts
export interface AxisMapping {
  column: string        // source column name
  displayName?: string  // optional display label override
}
```

### Updated `Widget`

Remove `dataSource?: DataSource`. Add:

```ts
dataSourceId?: string                           // references GlobalDataSource.id
dataSourceMapping?: Record<string, AxisMapping> // fieldKey ‚Üí { column, displayName? }
```

### Updated `DashboardState`

`DashboardState` (in `store/dashboard.ts` ‚Äî there is no separate `Dashboard` interface) gains:

```ts
dataSources: GlobalDataSource[]  // default []
```

### Retired types

`DataSource` and `DataSourceType` are removed from `types/dashboard.ts`. `DataSource.mapping: Record<string, string>` is superseded by `AxisMapping`.

## Store Changes

### Updated `loadDashboard` signature

```ts
loadDashboard(dashboard: {
  name: string
  widgets: Widget[]
  layout: LayoutItem[]
  dataSources?: GlobalDataSource[]   // optional for backwards compat
}): void
```

`dataSources` defaults to `[]` if absent (legacy dashboards). Migration logic runs after loading (see below).

### New actions in `store/dashboard.ts`

```ts
addGlobalDataSource(ds: GlobalDataSource): void
updateGlobalDataSource(id: string, patch: Partial<GlobalDataSource>): void
removeGlobalDataSource(id: string): void
  // clears dataSourceId AND dataSourceMapping on any widget referencing this id
updateWidgetDataSourceId(widgetId: string, dataSourceId: string | undefined): void
updateWidgetMapping(widgetId: string, mapping: Record<string, AxisMapping>): void
```

Remove existing `updateDataSource(widgetId, dataSource)`.

### Updated `saveDashboard`

Two changes required:

1. **Serialized body** ‚Äî add `dataSources` to the single `body` variable that is constructed once and reused for both the initial save attempt and the 404-fallback re-try:
   ```ts
   const body = JSON.stringify({ name, widgets: sanitizedWidgets, layout, dataSources })
   ```
   Without this, global data sources are not persisted and are lost on reload.

2. **CSV size guard** ‚Äî the existing per-widget `dataSource.type === 'csv'` check moves to the global sources array:
   - **Error stripping**: `GlobalDataSource` has no `error` field ‚Äî no stripping needed on widgets.
   - **Size guard**: Sum `JSON.stringify(ds.data ?? []).length` across all `dataSources` where `ds.type === 'csv'`. If sum > 3 MB, throw a descriptive error before saving.

### Migration on load

In `loadDashboard`, after setting state, run a one-time migration for each widget that still has the legacy `dataSource` field:

1. Build a `GlobalDataSource` from it:
   - `id`: `nanoid()`
   - `name`: `dataSource.fileName` ?? URL hostname of `dataSource.url` ?? `"Imported Source"`
   - Copy `type`, `data`, `fileName`, `url`, `gid`, `refreshInterval`
2. Deduplicate: if a `GlobalDataSource` with the same composite key already exists in `dataSources`, reuse its `id` instead of creating a new entry:
   - For Sheets: key is `(url, gid ?? '')` ‚Äî two widgets pointing to the same spreadsheet but different tabs are treated as distinct sources
   - For CSV: key is `fileName` (if present)
3. Convert `dataSource.mapping: Record<string, string>` ‚Üí `Record<string, AxisMapping>`:
   - `{ [fieldKey]: colName }` ‚Üí `{ [fieldKey]: { column: colName } }`
4. Set `widget.dataSourceId` and `widget.dataSourceMapping`, delete `widget.dataSource`.

Migration is silent and transparent ‚Äî users with v2.2 dashboards see no disruption.

## Toolbar Data Sources Panel

### Entry point

A **"Data Sources"** button in the existing top toolbar. When one or more sources are connected, a count badge appears (e.g. "2"). Clicking toggles a full-width slide-down panel below the toolbar ‚Äî no modal.

### Panel zones

**Zone 1 ‚Äî Connected sources** (visible when `dataSources.length > 0`):

Each row shows: source name, type badge (CSV / Sheet), row count, rename icon, delete icon. Clicking a row expands it inline to show reconnect / replace options (same CSV/Sheets flow). Deleting a source that is referenced by widgets shows a confirmation: "X widgets use this source ‚Äî they will be disconnected."

**Zone 2 ‚Äî Add source:**

"Upload CSV" and "Connect Google Sheet" buttons trigger the existing CSV/Sheets connection flow (file picker ‚Üí preview table ‚Üí confirm). On confirm, a `GlobalDataSource` is created and pushed to `dashboard.dataSources`. The source is auto-named from the filename or sheet hostname; the user can rename inline.

**Zone 3 ‚Äî AI Generate** (visible when `dataSources.length > 0`):

- Prompt input field: placeholder `"What would you like to visualize?"`
- **Generate** button: calls `POST /api/ai/generate-widgets`, appends returned widgets to canvas
- **Auto-suggest**: when the first source is connected and the canvas has **no widgets**, the generate endpoint fires automatically with the default prompt `"Generate a dashboard for this data"`. This counts against the user's generation limit identically to a manual generation ‚Äî no special bypass.
- Loading state: "Generating widgets‚Ä¶" shown in Zone 3 while request is in flight

## Per-Widget Mapping (Right Panel)

The existing "Data Source" section is replaced. Rendering adapts to three states:

### State 1 ‚Äî No global sources

```
Connect a data source from the toolbar to get started.
```

### State 2 ‚Äî Sources exist, none assigned to this widget

A dropdown listing all `dashboard.dataSources` by name. Selecting one calls `updateWidgetDataSourceId` and immediately triggers AI auto-map.

### State 3 ‚Äî Source assigned

- Source name shown with a **"change"** link that reopens the dropdown
- One mapping row per field in `widgetFieldRegistry[widget.type]`:
  - Field label (e.g. "X Axis", "Y Axis", "Value", "Name")
  - Column picker: all columns from the selected `GlobalDataSource`
  - Display name input: optional, placeholder = raw column name
- **Auto-map with AI** button ‚Äî calls `/api/ai/map-columns` (unchanged endpoint; requires authentication ‚Äî unauthenticated users will receive a 401 and the button is a no-op for them, same as v2.2). The `Record<string, string>` response is converted at the call site to `Record<string, AxisMapping>` by wrapping each value: `{ [k]: { column: v } }`. Result is passed to `updateWidgetMapping`.
- **Disconnect** button: clears `dataSourceId` and `dataSourceMapping`

### Hook + mapping updates

`useDataSource(widget)` is updated to resolve the data source from the store instead of reading `widget.dataSource` directly:

```ts
export function useDataSource(widget: Widget): DataSourceResult {
  const dataSources = useDashboardStore((s) => s.dataSources)
  const ds = dataSources.find((d) => d.id === widget.dataSourceId) ?? null
  // ... hook body using ds instead of widget.dataSource
}
```

**Restructuring notes**:

- Initialise all state with empty defaults: `const [rows, setRows] = useState<...>(null)`, `const [isLoading, setIsLoading] = useState(false)`, etc. Do not derive initial values from `ds` at `useState` call time.
- **Remove the early-return guard** (`if (!ds) return { rows: null, ... }`) that currently sits after all hooks. Instead, handle `ds === null` inside the `useEffect`: when `ds` is null, call `setRows(null)` and return ‚Äî this keeps all hook calls unconditional (no Rules of Hooks violation).
- For CSV sources (`ds.type === 'csv'`), set rows synchronously inside the `useEffect` via `setRows(ds.data ?? [])` rather than as an `useState` initial value.
- The `useEffect` dependency array uses `ds?.type`, `ds?.url`, `ds?.gid`, `ds?.refreshInterval`, `ds?.data` ‚Äî these now come from the looked-up `GlobalDataSource`.

The return type (`{ rows, isLoading, error, lastFetched }`) is unchanged.

`applyMapping` signature changes:

```ts
// OLD
function applyMapping(
  fieldKey: string,
  fieldType: string,
  rows: Record<string, unknown>[],
  mapping: Record<string, string>
): unknown | null

// NEW
function applyMapping(
  fieldKey: string,
  fieldType: string,
  rows: Record<string, unknown>[],
  mapping: Record<string, AxisMapping>
): unknown | null
```

Internal changes required in `lib/apply-mapping.ts`:

- **All `mapping[fieldKey]` reads**: replace the raw string with `mapping[fieldKey]?.column` (e.g. `const col = mapping[fieldKey]?.column`)
- **Pass-through path** (`fieldKey === 'rows'` / `PASS_THROUGH_FIELD`): `Object.values(mapping)` now returns `AxisMapping` objects ‚Äî extract column names: `const mappedCols = new Set(Object.values(mapping).map((m) => m.column))`
- **Rename path** (array-of-objects): `for (const [widgetKey, axisMapping] of Object.entries(mapping))` then `out[widgetKey] = row[axisMapping.column] ?? null`
- **activity-feed `id` guard**: `if (fieldKey === 'events' && !out['id']) out['id'] = String(i)` ‚Äî unchanged

`WidgetRenderer.tsx` must also be updated:

- **Guard**: change `widget.dataSource?.mapping` ‚Üí `widget.dataSourceMapping`
- **Mapping variable**: `const mapping = widget.dataSourceMapping` (type is now `Record<string, AxisMapping>`)
- **data-table column synthesis** (currently line 42): update to use `AxisMapping`:
  ```ts
  patch['columns'] = Object.entries(mapping).map(([, axisMapping]) => ({
    key: axisMapping.column,
    label: axisMapping.displayName ?? axisMapping.column,
    sortable: true,
  }))
  ```
- **Display names for chart widgets**: after building `patch`, if `CHART_TYPES.has(widget.type)`, optionally read `mapping['data']?.displayName` to set a series label ‚Äî this is a best-effort enhancement, not required for correctness.

Display names are **not** threaded through `applyMapping` itself ‚Äî they are applied in `WidgetRenderer` as described above. Widget types that have no axis label config (e.g. `kpi`, `gauge`) ignore `displayName`.

## AI Widget Generation

### Endpoint: `POST /api/ai/generate-widgets`

**Auth:** Same as `/api/ai/generate` ‚Äî no authentication required; rate-limited by `isGenerationLimitReached(ip, userId)` where `ip` is extracted from request headers and `userId` from the Clerk session (if present). Returns `429` when limit is reached.

**Request:**
```ts
{
  dataSourceId: string
  columns: string[]
  sampleRows: Record<string, unknown>[]   // first 5 rows
  prompt: string
}
```

**Response:**
```ts
{
  widgets: Array<{
    type: WidgetType
    config: Partial<WidgetConfig>          // AI fills known fields; missing fields get widget defaults
    dataSourceId: string
    dataSourceMapping: Record<string, AxisMapping>
    title?: string
  }>
}
```

**System prompt constraints:**
- Output only valid `WidgetType` values from this list: `line-chart`, `area-chart`, `bar-chart`, `donut-chart`, `funnel-chart`, `kpi`, `data-table`, `text-note`, `progress-tracker`, `activity-feed`, `gauge`
- `dataSourceMapping` column values must exist verbatim in the `columns` input array
- Generate 3‚Äì6 widgets; include a `data-table` when column count > 3
- Vary chart types based on data shape: time-series columns ‚Üí `line-chart` or `area-chart`; categorical ‚Üí `bar-chart` or `donut-chart`; single numeric ‚Üí `kpi`
- Return valid JSON only ‚Äî no prose, no markdown wrapper

**Response validation (server-side):**
- Verify each `type` is a valid `WidgetType`
- Verify each `dataSourceMapping` column value exists in the request `columns` array
- Strip any widget that fails validation rather than returning an error (partial success is acceptable)
- Merge returned `config` with widget defaults (`defaultConfig[type]`) so the renderer always receives a complete config

**Placement:** The existing `addWidget(type)` action does not accept positional arguments. Add a new store action:

```ts
addGeneratedWidgets(generated: Array<{
  type: WidgetType
  config: Partial<WidgetConfig>
  dataSourceId: string
  dataSourceMapping: Record<string, AxisMapping>
}>): void
```

This action computes layout for all incoming widgets in one atomic `set()` call. Widget placement respects full-width types:

**Pre-pass: classify each widget**
- If `widgetRegistry[type].defaultSize.w === 12` ‚Üí full-width widget (placed alone on its own row, `w: 12`)
- Otherwise ‚Üí half-width widget (`w: 6`, placed in pairs)

**Layout algorithm** (processing widgets in order):
1. `curY` = current max `y + h` across existing layout items (or 0 if empty)
2. Maintain a pending half-width slot (`leftWidget` buffer, initially empty)
3. For each widget:
   - If **full-width**: flush any pending `leftWidget` first (place it at `x: 0, y: curY, w: 6`; advance `curY += leftWidget.h`; clear buffer), then place full-width widget at `x: 0, y: curY, w: 12, h: registryH`; advance `curY += registryH`
   - If **half-width** and buffer is empty: store in `leftWidget` buffer
   - If **half-width** and buffer has a widget: place left at `x: 0, y: curY, w: 6, h: leftH`; place right at `x: 6, y: curY, w: 6, h: rightH`; advance `curY += Math.max(leftH, rightH)`; clear buffer
4. After all widgets, flush any remaining `leftWidget` (place at `x: 0, y: curY, w: 6`)

**Heights**: always use `widgetRegistry[type].defaultSize.h` ‚Äî do not override heights regardless of type.

Each widget is initialised with `{ ...widgetRegistry[type].defaultConfig, ...config }` so missing AI-provided fields fall back to defaults.

## Migration Compatibility

| v2.2 feature | v2.3 status |
|---|---|
| Per-widget CSV upload | Migrated to global source on load |
| Per-widget Google Sheets | Migrated to global source on load |
| `dataSource.mapping` (fieldKey ‚Üí colName) | Converted to `AxisMapping` (fieldKey ‚Üí { column }) |
| Auto-map with AI | Preserved in right panel; response adapted to `AxisMapping` at call site |
| Column mapping dropdowns | Replaced by axis-aware rows with display name input |
| `saveDashboard` CSV size guard | Updated to sum across `dataSources` instead of widgets |
| `saveDashboard` body | `dataSources` added to serialized JSON |
| `addWidget` for AI generation | New `addGeneratedWidgets` action computes 2-column grid layout |

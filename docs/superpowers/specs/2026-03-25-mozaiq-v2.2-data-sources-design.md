# Mozaiq v2.2 — Data Sources Design

## Goal

Connect any widget to real data via CSV upload (snapshot) or Google Sheets (live polling). Users can map columns to widget fields manually or use AI to auto-map. The `DataSource` type is designed as a first-class object so future storage backends (Vercel Blob, separate DB table) can be swapped in without touching widget or UI code.

## Architecture

```
CSV upload  → papaparse (client-side) → rows stored in widget.dataSource.data (Postgres JSON)
Sheet URL   → /api/data/google-sheets → fetch public CSV export → parse → return rows
                                                    ↓
                              useDataSource(widget) hook
                              ├── CSV: returns widget.dataSource.data directly
                              └── Google Sheets: fetches via route, polls every N seconds
                                                    ↓
                              WidgetRenderer calls useDataSource(widget),
                              passes resolvedRows + mapping to applyMapping(),
                              passes resolvedData to the widget component
                                                    ↓
                              Widget renderer reads resolvedData (falls back to mock config data)
```

Both source types share the same `mapping` object (`widgetField → columnName`). Widget renderers receive already-mapped rows via the `resolvedData` prop — they don't know or care whether the data came from CSV or Sheets. The `applyMapping` utility (see below) handles the transformation centrally so each renderer doesn't implement its own version.

## Tech Stack

- **papaparse** — CSV parsing (client-safe, no Node.js dependency; always pass `worker: false` on the server)
- **Vercel AI SDK + AI Gateway** — map-columns endpoint
- **Next.js Route Handler** — Google Sheets proxy (avoids CORS on public export URLs)
- Existing: Prisma/Neon (dashboard JSON), `@clerk/nextjs` (auth), Zustand (store)

## Type Changes — `types/dashboard.ts`

Replace the existing `DataSource` scaffold with:

```ts
export type DataSourceType = 'csv' | 'google-sheets'

export interface DataSource {
  type: DataSourceType
  // CSV snapshot
  data?: Record<string, unknown>[]
  fileName?: string
  // Google Sheets live
  url?: string
  gid?: string              // optional numeric sheet tab ID for multi-tab sheets
  refreshInterval?: number  // seconds; default 60, minimum 10; enforced in useDataSource
  // Shared
  mapping?: Record<string, string>  // widgetField → columnName
  error?: string                    // last fetch/parse error message, not persisted
}
```

`Widget.dataSource` stays optional — widgets without a connected source continue to use mock config data.

**Backward compatibility:** The existing `'mock'` and `'api'` type values were scaffold-only and never written to the database. No migration is needed. If a widget's `dataSource.type` is read and doesn't match `'csv' | 'google-sheets'`, treat it as no source and fall back to mock config data.

**Note:** `lastFetched` is intentionally omitted from the persisted `DataSource` type — it is transient polling metadata and is kept only in `useDataSource` local state. `DataSourcePanel` reads `lastFetched` from the hook return value, not from the store.

## New Utility — `lib/apply-mapping.ts` (new)

Central transformation utility called by `WidgetRenderer` before passing data to widget components. This prevents 11 renderers from each implementing their own mapping logic.

```ts
/**
 * Transforms raw source rows into the shape a specific widget field expects,
 * using the column mapping from widget.dataSource.mapping.
 *
 * fieldKey:       the widget field being resolved (e.g. 'data', 'value', 'rows')
 * fieldType:      from widgetFieldRegistry — determines output shape
 * rows:           raw rows from useDataSource
 * mapping:        widgetField → columnName from dataSource.mapping
 *
 * Returns null if the field is not mapped or rows is empty.
 */
export function applyMapping(
  fieldKey: string,
  fieldType: WidgetFieldDef['type'],
  rows: Record<string, unknown>[],
  mapping: Record<string, string>,
): unknown | null
```

Behaviour by field type:
- **`'number'`** — returns `Number(rows[0][mapping[fieldKey]])` from the first row; `null` if column missing or not numeric
- **`'string'`** — returns `String(rows[0][mapping[fieldKey]])` from the first row; `null` if column missing
- **`'array-of-objects'`** — returns rows mapped through the column mapping. For chart widgets the mapping should include `name` → some column and `value` → some column; the function renames columns accordingly. For table/activity/progress it passes rows through with column names preserved (the mapping defines which columns to include).

`WidgetRenderer` calls `applyMapping` for each field defined in `widgetFieldRegistry[widget.type]` and passes the results as `resolvedData` to the component.

## New Hook — `hooks/useDataSource.ts`

Resolves a widget's data source into rows. CSV sources return immediately (data is inline). Google Sheets sources fetch via the proxy route and poll on `refreshInterval`.

```ts
export function useDataSource(widget: Widget): {
  rows: Record<string, unknown>[] | null
  isLoading: boolean
  error: string | null
  lastFetched: string | null  // ISO timestamp, local state only — not written to store
}
```

- Returns `null` rows if `widget.dataSource` is not set
- On Google Sheets: fetches `/api/data/google-sheets?url=<encoded>&gid=<gid>` on mount and every `refreshInterval` seconds (default 60s, minimum 10s enforced here)
- `refreshInterval` of `0` means no automatic polling — user must reconnect to refresh (this is the "manual" option in the UI)
- Caches last successful rows in local state; on fetch error, keeps stale rows and sets `error`
- `lastFetched` is local state only — never written to the Zustand store

## New File — `lib/parse-csv.ts`

Thin wrapper around papaparse for consistent config. Used both client-side (file upload) and server-side (proxy route — always pass `worker: false` on the server).

```ts
export function parseCsvText(text: string): {
  columns: string[]
  rows: Record<string, unknown>[]
  error?: string
  warning?: string  // present when rows were truncated
}
```

- Uses `header: true`, `dynamicTyping: true`, `skipEmptyLines: true`, `worker: false`
- Returns columns (header row keys) and typed rows
- Max 10,000 rows enforced — truncates and sets `warning: 'File truncated to 10,000 rows'`

## New Route — `app/api/data/google-sheets/route.ts`

Server proxy to avoid CORS on Google Sheets public CSV export URLs.

```ts
// GET /api/data/google-sheets?url=<encoded-sheet-url>&gid=<optional-tab-id>
export async function GET(req: Request)
```

- Accepts a full Google Sheets URL (extracts spreadsheet ID via regex) or a bare spreadsheet ID
- Constructs export URL: `https://docs.google.com/spreadsheets/d/{id}/export?format=csv[&gid={gid}]`
- 5s fetch timeout via `AbortController`
- Passes response text through `parseCsvText` (with `worker: false`)
- Returns `{ columns, rows, warning? }`
- Returns 400 if URL is not a recognisable Google Sheets URL
- Returns 502 if upstream fetch fails, times out, or sheet is not public
- No auth required — public sheets only

## New Route — `app/api/ai/map-columns/route.ts`

Given a widget type, available columns, and sample rows, returns a suggested mapping.

```ts
// POST /api/ai/map-columns
// Body: { widgetType: WidgetType, columns: string[], sampleRows: Record<string, unknown>[], widgetFields: WidgetFieldDef[] }
// Response: { mapping: Record<string, string> }
```

- Requires Clerk auth — returns 401 if unauthenticated. Not rate-limited or logged in `GenerationLog`.
- Uses `generateText` via AI Gateway (`anthropic/claude-haiku-4-5-20251001` — fast + cheap for this task)
- System prompt instructs the model to return only a JSON object mapping widgetField keys to column name strings
- Validates response is a plain object with string values; returns 500 if model returns unparseable output
- Returns 400 if required body fields are missing

## New Component — `components/builder/DataSourcePanel.tsx`

Replaces the "Live data connections available in v2" placeholder in RightPanel. Rendered inside the Data section when a widget is selected.

### States

**No source connected:**
- Two buttons: "Upload CSV" and "Connect Google Sheet"
- One-line description under each: "Snapshot — data stored with your dashboard" / "Live — syncs from your public sheet"

**CSV flow:**
1. File input (`accept=".csv"`) → `parseCsvText` client-side → preview (column names + first 3 rows)
2. Shows `warning` banner if rows were truncated
3. "Use this data" button → calls `updateDataSource(widgetId, { type: 'csv', data: rows, fileName })`
4. Error state if `parseCsvText` returns an `error`

**Google Sheets flow:**
1. URL input + "Connect" button
2. Fetches via `/api/data/google-sheets` → preview (column names + first 3 rows)
3. Optional gid input for multi-tab sheets (labelled "Sheet tab ID — leave blank for first tab")
4. Refresh interval picker: 10s / 30s / 1 min / 5 min / 15 min / Manual (0 = no auto-poll)
5. "Connect" button → calls `updateDataSource(widgetId, { type: 'google-sheets', url, gid?, refreshInterval })`
6. Error state if sheet not public or URL unrecognised
7. "Last synced: Xs ago" — reads `lastFetched` from `useDataSource` hook return (not from store)

**Source connected (either type):**
- Source badge (CSV filename or sheet URL hostname)
- **Column mapping section** — one row per field in `widgetFieldRegistry[widget.type]`:
  - Field label + dropdown of available columns + "(none)" option
  - Required fields with no mapping highlighted in amber
- **"Auto-map with AI"** button — calls `/api/ai/map-columns`, fills dropdowns; user can adjust
- **"Disconnect"** link — calls `updateDataSource(widgetId, undefined)`, reverts to mock data

**Data table column display:** When `resolvedData` is active, the table renders columns derived from the mapping values (the raw source column names), with each mapping key as the display label. Sortability defaults to `true` for all auto-derived columns.

## Widget Renderer Changes

### `WidgetProps` — `types/dashboard.ts`

```ts
export interface WidgetProps {
  config: WidgetConfig
  isSelected: boolean
  onSelect: () => void
  onDelete: () => void
  isReadOnly?: boolean
  resolvedData?: Record<string, string | number | unknown>[] | null  // null = use mock
}
```

### `WidgetRenderer` — `components/widgets/WidgetRenderer.tsx`

`WidgetRenderer` (not `WidgetWrapper`) is the call site for `useDataSource`. It receives the full `Widget` object, calls `useDataSource(widget)`, runs `applyMapping` for each field in `widgetFieldRegistry[widget.type]`, and passes the resulting `resolvedData` to the component. `WidgetWrapper` remains purely presentational (border, drag handle, delete button).

For each widget type, `WidgetRenderer` builds `resolvedData` as follows:

**Chart widgets** (line, area, bar, donut, funnel): `applyMapping('data', 'array-of-objects', rows, mapping)` → `{ name, value }[]`. Falls back to `c.data` then mock.

**KPI**: `applyMapping('value', 'number', rows, mapping)` → single number for `value`; same for `change`. Falls back to `c.value` / `c.change`.

**Data table**: `applyMapping('rows', 'array-of-objects', rows, mapping)` → array of row objects. Columns are derived from mapping values (see DataSourcePanel section). Falls back to `c.rows`.

**Progress tracker**: `applyMapping('items', 'array-of-objects', rows, mapping)` → `{ label, value, max }[]`.

**Activity feed**: `applyMapping('events', 'array-of-objects', rows, mapping)` → `{ id, label, time }[]`.

**Text note**: `applyMapping('content', 'string', rows, mapping)` → string from first row. Falls back to `c.content`.

**Gauge**: `applyMapping('value', 'number', rows, mapping)` → number from first row. Falls back to `c.value`.

## Store Changes — `store/dashboard.ts`

Add two actions:

```ts
// Used by DataSourcePanel on connect/disconnect — sets isDirty: true
updateDataSource: (widgetId: string, dataSource: DataSource | undefined) => void

// Used internally if needed for non-config writes; currently unused (lastFetched is hook-local)
// Included here to document the intentional design decision.
```

Only `updateDataSource` is implemented. `lastFetched` is never written to the store, so polls never trigger `isDirty`.

## Error Handling

| Scenario | Behaviour |
|----------|-----------|
| CSV too large (>10K rows) | `parseCsvText` truncates, sets `warning`; DataSourcePanel shows warning banner; source still usable |
| CSV parse error | `error` field set; DataSourcePanel shows error message; source not saved |
| Google Sheet not public | 502 from proxy → "Sheet not found or not public" in panel |
| Google Sheet fetch timeout (5s) | 502 → stale rows kept in hook state; `error` shown in DataSourcePanel |
| AI map-columns failure | Toast "Auto-map failed — map manually"; mapping dropdowns unchanged |
| Column mapped but missing in rows | `applyMapping` returns `null` for that field; widget falls back to config value or renders empty/zero — no crash |
| Unknown dataSource.type in stored data | Treated as no source; widget falls back to mock config data |

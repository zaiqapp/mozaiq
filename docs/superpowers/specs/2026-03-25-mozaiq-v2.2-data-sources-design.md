# Mozaiq v2.2 â€” Data Sources Design

## Goal

Connect any widget to real data via CSV upload (snapshot) or Google Sheets (live polling). Users can map columns to widget fields manually or use AI to auto-map. The `DataSource` type is designed as a first-class object so future storage backends (Vercel Blob, separate DB table) can be swapped in without touching widget or UI code.

## Architecture

```
CSV upload  â†’ papaparse (client-side) â†’ rows stored in widget.dataSource.data (Postgres JSON)
Sheet URL   â†’ /api/data/google-sheets â†’ fetch public CSV export â†’ parse â†’ return rows
                                                    â†“
                              useDataSource(widget) hook
                              â”śâ”€â”€ CSV: returns widget.dataSource.data directly
                              â””â”€â”€ Google Sheets: fetches via route, polls every N seconds
                                                    â†“
                              WidgetRenderer calls useDataSource(widget),
                              passes resolvedRows + mapping to applyMapping(),
                              passes resolvedData to the widget component
                                                    â†“
                              Widget renderer reads resolvedData (falls back to mock config data)
```

Both source types share the same `mapping` object (`widgetField â†’ columnName`). Widget renderers receive already-mapped rows via the `resolvedData` prop â€” they don't know or care whether the data came from CSV or Sheets. The `applyMapping` utility (see below) handles the transformation centrally so each renderer doesn't implement its own version.

## Tech Stack

- **papaparse** â€” CSV parsing (client-safe, no Node.js dependency; always pass `worker: false` on the server)
- **Vercel AI SDK + AI Gateway** â€” map-columns endpoint
- **Next.js Route Handler** â€” Google Sheets proxy (avoids CORS on public export URLs)
- Existing: Prisma/Neon (dashboard JSON), `@clerk/nextjs` (auth), Zustand (store)

## Type Changes â€” `types/dashboard.ts`

Replace the existing `DataSource` scaffold with:

```ts
export type DataSourceType = 'csv' | 'google-sheets'

export interface DataSource {
  type: DataSourceType
  // CSV snapshot
  data?: Record<string, unknown>[]
  fileName?: string
  // Google Sheets live
  url?: string
  gid?: string              // optional numeric sheet tab ID for multi-tab sheets
  refreshInterval?: number  // seconds; default 60, minimum 10; enforced in useDataSource
  // Shared
  mapping?: Record<string, string>  // widgetField â†’ columnName
  error?: string                    // last fetch/parse error message, not persisted
}
```

`Widget.dataSource` stays optional â€” widgets without a connected source continue to use mock config data.

**Backward compatibility:** The existing `'mock'` and `'api'` type values were scaffold-only and never written to the database. No migration is needed. If a widget's `dataSource.type` is read and doesn't match `'csv' | 'google-sheets'`, treat it as no source and fall back to mock config data.

**Note:** `lastFetched` is intentionally omitted from the persisted `DataSource` type â€” it is transient polling metadata and is kept only in `useDataSource` local state. `DataSourcePanel` reads `lastFetched` from the hook return value, not from the store.

## New Utility â€” `lib/apply-mapping.ts` (new)

Central transformation utility called by `WidgetRenderer` before passing data to widget components. This prevents 11 renderers from each implementing their own mapping logic.

```ts
/**
 * Transforms raw source rows into the shape a specific widget field expects.
 *
 * fieldKey:       the widget field being resolved (e.g. 'data', 'value', 'rows')
 * fieldType:      from widgetFieldRegistry â€” determines output shape
 * rows:           raw rows from useDataSource
 * mapping:        widgetField â†’ columnName from dataSource.mapping
 *
 * Returns null if the field is not mapped, rows is empty, or the value is missing.
 */
export function applyMapping(
  fieldKey: string,
  fieldType: WidgetFieldDef['type'],
  rows: Record<string, unknown>[],
  mapping: Record<string, string>,
): unknown | null
```

**Behaviour by field type:**

- **`'number'`** â€” returns `Number(rows[0][mapping[fieldKey]])` from the first row; `null` if column not in mapping, value missing, or result is `NaN`
- **`'string'`** â€” returns `String(rows[0][mapping[fieldKey]])` from the first row; `null` if column not in mapping or value missing
- **`'array-of-objects'`** â€” two sub-modes depending on widget type:

  **Rename mode** (chart widgets, funnel, progress-tracker, activity-feed): the mapping entries are `widgetKey â†’ sourceColumn`. Output rows have the widget's expected key names. For example, mapping `{ name: 'month', value: 'revenue' }` transforms `[{ month: 'Jan', revenue: 120 }]` â†’ `[{ name: 'Jan', value: 120 }]`. This ensures the widget accesses fixed keys (`name`, `value` for charts; `label`, `value`, `max` for progress-tracker; `id`, `label`, `time` for activity-feed â€” `id` defaults to row index if not mapped).

  **Pass-through mode** (data-table only): rows are passed through with source column names preserved. The mapping selects which columns to include (only mapped columns appear in output rows). `WidgetRenderer` also synthesises a `columns` array from the mapping: `Object.entries(mapping).map(([widgetKey, sourceCol]) => ({ key: sourceCol, label: widgetKey, sortable: true }))` â€” this synthesised `columns` replaces `c.columns` in the merged config when live data is active.

`WidgetRenderer` calls `applyMapping` for each field in `widgetFieldRegistry[widget.type]`, then **merges the resolved values back into the `config` object** before passing it as `config` to the widget component (Path A â€” no new prop). This means no renderer changes are needed beyond their existing mock-fallback guards (e.g. `c.data && c.data.length > 0`). For chart widgets, `WidgetRenderer` also sets `dataKey: 'value'` in the merged config when live data is active, since `applyMapping` always normalises chart rows to `{ name, value }[]`.

## New Hook â€” `hooks/useDataSource.ts`

Resolves a widget's data source into rows. CSV sources return immediately (data is inline). Google Sheets sources fetch via the proxy route and poll on `refreshInterval`.

```ts
export function useDataSource(widget: Widget): {
  rows: Record<string, unknown>[] | null
  isLoading: boolean
  error: string | null
  lastFetched: string | null  // ISO timestamp, local state only â€” not written to store
}
```

- Returns `null` rows if `widget.dataSource` is not set
- On Google Sheets: fetches `/api/data/google-sheets?url=<encoded>&gid=<gid>` on mount and every `refreshInterval` seconds (default 60s)
- The 10s minimum is enforced only when `refreshInterval > 0`; a value of `0` bypasses polling entirely (manual mode â€” no auto-poll, no clamping)
- Caches last successful rows in local state; on fetch error, keeps stale rows and sets `error`
- `lastFetched` is local state only â€” never written to the Zustand store

## New File â€” `lib/parse-csv.ts`

Thin wrapper around papaparse for consistent config. Used both client-side (file upload) and server-side (proxy route â€” always pass `worker: false` on the server).

```ts
export function parseCsvText(text: string): {
  columns: string[]
  rows: Record<string, unknown>[]
  error?: string
  warning?: string  // present when rows were truncated
}
```

- Uses `header: true`, `dynamicTyping: true`, `skipEmptyLines: true`, `worker: false`
- Returns columns (header row keys) and typed rows
- Max 10,000 rows enforced â€” truncates and sets `warning: 'File truncated to 10,000 rows'`

## New Route â€” `app/api/data/google-sheets/route.ts`

Server proxy to avoid CORS on Google Sheets public CSV export URLs.

```ts
// GET /api/data/google-sheets?url=<encoded-sheet-url>&gid=<optional-tab-id>
export async function GET(req: Request)
```

- Accepts a full Google Sheets URL (extracts spreadsheet ID via regex) or a bare spreadsheet ID
- Constructs export URL: `https://docs.google.com/spreadsheets/d/{id}/export?format=csv[&gid={gid}]`
- 5s fetch timeout via `AbortController`
- Passes response text through `parseCsvText` (with `worker: false`)
- Returns `{ columns, rows, warning? }`
- Returns 400 if URL is not a recognisable Google Sheets URL
- Returns 502 if upstream fetch fails, times out, or sheet is not public
- No auth required â€” public sheets only

## New Route â€” `app/api/ai/map-columns/route.ts`

Given a widget type, available columns, and sample rows, returns a suggested mapping.

```ts
// POST /api/ai/map-columns
// Body: { widgetType: WidgetType, columns: string[], sampleRows: Record<string, unknown>[], widgetFields: WidgetFieldDef[] }
// Response: { mapping: Record<string, string> }
```

- Requires Clerk auth â€” returns 401 if unauthenticated. Not rate-limited or logged in `GenerationLog`.
- Uses `generateText` via AI Gateway (`anthropic/claude-haiku-4-5-20251001` â€” fast + cheap for this task; verify this model ID is live in the AI Gateway before implementation)
- System prompt instructs the model to return only a JSON object mapping widgetField keys to column name strings
- Validates response is a plain object with string values; returns 500 if model returns unparseable output
- Returns 400 if required body fields are missing

## New Component â€” `components/builder/DataSourcePanel.tsx`

Replaces the "Live data connections available in v2" placeholder in RightPanel. Rendered inside the Data section when a widget is selected.

### States

**No source connected:**
- Two buttons: "Upload CSV" and "Connect Google Sheet"
- One-line description under each: "Snapshot â€” data stored with your dashboard" / "Live â€” syncs from your public sheet"

**CSV flow:**
1. File input (`accept=".csv"`) â†’ `parseCsvText` client-side â†’ preview (column names + first 3 rows)
2. Shows `warning` banner if rows were truncated
3. "Use this data" button â†’ calls `updateDataSource(widgetId, { type: 'csv', data: rows, fileName })`
4. Error state if `parseCsvText` returns an `error`

**Google Sheets flow:**
1. URL input + "Connect" button
2. Fetches via `/api/data/google-sheets` â†’ preview (column names + first 3 rows)
3. Optional gid input for multi-tab sheets (labelled "Sheet tab ID â€” leave blank for first tab")
4. Refresh interval picker: 10s / 30s / 1 min / 5 min / 15 min / Manual (0 = no auto-poll)
5. "Connect" button â†’ calls `updateDataSource(widgetId, { type: 'google-sheets', url, gid?, refreshInterval })`
6. Error state if sheet not public or URL unrecognised
7. "Last synced: Xs ago" â€” reads `lastFetched` from `useDataSource` hook return (not from store)

**Source connected (either type):**
- Source badge (CSV filename or sheet URL hostname)
- **Column mapping section** â€” one row per field in `widgetFieldRegistry[widget.type]`:
  - Field label + dropdown of available columns + "(none)" option
  - Required fields with no mapping highlighted in amber
- **"Auto-map with AI"** button â€” calls `/api/ai/map-columns`, fills dropdowns; user can adjust
- **"Disconnect"** link â€” calls `updateDataSource(widgetId, undefined)`, reverts to mock data

**Data table column display:** When `resolvedData` is active, the table renders columns derived from the mapping values (the raw source column names), with each mapping key as the display label. Sortability defaults to `true` for all auto-derived columns.

## Widget Renderer Changes

### `WidgetProps` â€” no change needed

`WidgetProps` is unchanged. `WidgetRenderer` merges resolved values back into the `config` object (Path A) so all renderers continue reading only `config`. No new prop is added.

### `WidgetRenderer` â€” `components/widgets/WidgetRenderer.tsx`

`WidgetRenderer` (not `WidgetWrapper`) is the call site for `useDataSource`. It receives the full `Widget` object, calls `useDataSource(widget)`, and â€” if `rows` is non-null â€” calls `applyMapping` for each field in `widgetFieldRegistry[widget.type]` and merges the results into a patched config before passing it to the component. `WidgetWrapper` remains purely presentational.

```ts
// Pseudocode in WidgetRenderer
const { rows } = useDataSource(widget)
let patchedConfig = widget.config

if (rows && widget.dataSource?.mapping) {
  const mapping = widget.dataSource.mapping
  const fields = widgetFieldRegistry[widget.type]
  const patch: Record<string, unknown> = {}

  for (const field of fields) {
    const resolved = applyMapping(field.key, field.type, rows, mapping)
    if (resolved !== null) patch[field.key] = resolved
  }

  // Chart widgets: fix dataKey to 'value' since applyMapping normalises to { name, value }
  if (isChartWidget(widget.type) && patch['data']) patch['dataKey'] = 'value'

  // Data table: synthesise columns from mapping when rows are live
  if (widget.type === 'data-table' && patch['rows']) {
    patch['columns'] = Object.entries(mapping).map(([widgetKey, sourceCol]) => ({
      key: sourceCol, label: widgetKey, sortable: true,
    }))
  }

  patchedConfig = { ...widget.config, ...patch }
}
```

**null fallback rule:** if `applyMapping` returns `null` for a field (column not mapped, value missing, or conversion failed), that field is not included in `patch` â€” the original config value is preserved unchanged. This means a KPI with an unmapped `change` field continues to show its config `change` value rather than `null`, avoiding a silent incorrect trend arrow.

**Per widget type â€” what gets patched:**

| Widget | Fields patched | Notes |
|--------|---------------|-------|
| line/area/bar/donut-chart | `data`, `dataKey` | `dataKey` always set to `'value'` when data resolved |
| funnel-chart | `data` | rename mode: `{ name, value }[]` |
| kpi | `value`, `change` | number fields; null = keep config value |
| gauge | `value` | number field |
| data-table | `rows`, `columns` | pass-through mode; columns synthesised |
| progress-tracker | `items` | rename mode: `{ label, value, max }[]` |
| activity-feed | `events` | rename mode: `{ id, label, time }[]`; id defaults to row index |
| text-note | `content` | string from first row's mapped column |

## Store Changes â€” `store/dashboard.ts`

Add one action:

```ts
// Used by DataSourcePanel on connect/disconnect â€” sets isDirty: true
updateDataSource: (widgetId: string, dataSource: DataSource | undefined) => void
```

`DataSource` must be imported from `@/types/dashboard` in the store file. `lastFetched` is never written to the store, so polls never trigger `isDirty`.

**Persisting `error`:** `DataSource.error` is runtime-only metadata. Before saving (in `saveDashboard`), strip the `error` field from any widget's `dataSource` to avoid persisting stale error strings. The simplest approach: in `saveDashboard`, map widgets through a function that omits `dataSource.error` from the JSON payload.

## Error Handling

| Scenario | Behaviour |
|----------|-----------|
| CSV too large (>10K rows) | `parseCsvText` truncates, sets `warning`; DataSourcePanel shows warning banner; source still usable |
| CSV parse error | `error` field set; DataSourcePanel shows error message; source not saved |
| Google Sheet not public | 502 from proxy â†’ "Sheet not found or not public" in panel |
| Google Sheet fetch timeout (5s) | 502 â†’ stale rows kept in hook state; `error` shown in DataSourcePanel |
| AI map-columns failure | Toast "Auto-map failed â€” map manually"; mapping dropdowns unchanged |
| Column mapped but missing in rows | `applyMapping` returns `null` for that field; widget falls back to config value or renders empty/zero â€” no crash |
| Unknown dataSource.type in stored data | Treated as no source; widget falls back to mock config data |
